# AI 使用反思报告

> Ommateum 项目开发过程中 AI 辅助的实践与总结

---

## 一、使用范围

本项目对 AI 的使用保持克制：**文档初稿和核心代码由人工完成，AI 仅参与格式美化、图表翻译和代码注释修缮。**

| 环节 | 人工完成 | AI 辅助 |
|---|---|---|
| 架构设计报告 | 大纲、技术决策描述、模块职责说明 | 表格对齐、编号整理、目录结构格式化 |
| 数据模型设计 | 字段定义、关联关系描述 | 表格排版、关系图 ASCII→PUML 翻译 |
| API 文档 | 端点列表、参数说明 | 格式统一 |
| 核心代码 (src/) | 全部模块逻辑 | 注释补全、docstring 格式统一 |
| PlantUML 图 | ASCII 草图 | ASCII 转 PUML 代码 |
| 问题排查 | 定位问题、判断根因 | 提供 YOLO/SAM2 参数背景知识 |

---

## 二、效果评估

### 2.1 有效的方面

**格式整理节省时间。** 手工写好的文档内容，AI 能快速完成表格对齐、编号重排、链接补全等机械操作，减少人工排版负担。

**ASCII 图转 PUML 基本可用。** 人工绘制的 ASCII 架构草图，AI 能将其翻译为 PlantUML 代码，虽需一定修正（去掉`!theme`、去除色值），但整体节省了手写 DSL 的时间。

**代码注释补全准确度尚可。** 对于已有代码，AI 生成的 docstring 和行内注释语义大致正确，人工复核后可直接采用。

### 2.2 不足的方面

**初次生成质量不稳定。** PlantUML 代码首次产出存在语法错误（无效指令、hex 色值），需要人工指出后修正。

**需求理解依赖精确指令。** AI 无法自主判断文档组织方式（嵌入 vs 独立），需人工明确给出"放到哪个文件、替换哪一段"的具体指令。

**技术推理不可全信。** 对 YOLO label 路径映射等问题的分析虽然逻辑自洽，但无法替代实际运行验证。

---

## 三、反思

### 3.1 文档初稿必须人工写

体验表明，AI 生成的文档初稿存在两个风险：一是会遗漏人工设计时的隐含意图（AI 只能看到代码，看不到想法）；二是格式产出看似规整，但内容密度和措辞准确性不如人工。**人工写内容、AI 修格式**是更合理的分工。

### 3.2 核心代码不应交由 AI 生成

AI 从零生成的代码存在 API 虚构风险——它可能调用不存在的方法或使用过时的参数名。核心逻辑必须人工编写，AI 只能做注释和格式的修饰性工作。

### 3.3 图表翻译是 AI 的甜点区

ASCII 草图 → PlantUML DSL 的翻译任务属于机械转换，不涉及创造性判断，正好落在 AI 的能力范围内。但产出需要人工复核语法。

### 3.4 指令粒度决定返工次数

最有效的方式是一次性说清最终期望（目录结构、文件命名、引用路径），避免"先做 A 再改 B 再回退到 A"的反复。

---

## 四、总结

本项目 AI 使用率较低，定位明确：**人写初稿和核心逻辑，AI 做格式整理和图表翻译**。在这个边界内 AI 的辅助效果可靠，超出此边界（让 AI 自行生成文档内容或业务代码）则不可控。
